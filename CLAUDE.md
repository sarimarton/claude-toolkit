# Claude Toolkit

- Changes only take effect after push — the menubar "⬆ Update available" item handles reinstall.
- **The user also keeps a tracked `~/.config` repo** (`github.com/<user>/.config`) holding their own home setup: `install.sh` (Homebrew casks, `defaults`, fonts, LaunchAgents), `swiftbar/` (the live plugin deploy dir), `tmux/`, `ghostty/`, `hammerspoon/`, a top-level `CLAUDE.md`, etc. Assume every user of this toolkit has one. It is a SEPARATE repo from the toolkit: the toolkit generates/owns the plugin *templates* and the update flow; the `.config` repo owns machine bootstrap and the user's hand-maintained dotfiles. When something is bootstrap/dotfile-shaped (e.g. pinning a Homebrew cask version, a `defaults write`, a font install) it belongs in `~/.config/install.sh`, not the toolkit — and may be committed there directly (it is the user's own source-of-truth, not a downstream consumer repo). When the user references "the .config repo" or asks about home setup, look under `~/.config`.
- **NEVER push to consumer/downstream repos as a shortcut.** This toolkit generates files into consumer repos (e.g. `.github/workflows/auto-dev.yml`). When fixing a bug observed in a consumer repo, the ONLY allowed flow is: fix the template here → commit → push → user runs the menubar Update + per-repo "Update auto-dev" cycle to propagate. NEVER bypass this by editing or `gh api`-patching the downstream repo directly, even if the toolkit update cycle feels slow or the downstream change is "obviously the same." Shortcutting the user's workflow corrupts their source-of-truth, conflicts with their later push, and makes RCA impossible.
- **Module shell utilities are declared in the manifest's `cli` field**, not just symlinked. When a module ships a user-facing command (like `crt`, `claude-tmux`, `ts`, `claude-ultraresume`), add a `cli: [{ name, description, script, usage? }]` entry so it surfaces under `claude-toolkit tools` (discovery) and `claude-toolkit run <name>` (delegated exec). `script` is the installed filename under the scripts dir. The module's own `~/.local/bin` symlink (in postInstall) stays — `cli` is for aggregation/discoverability, not a replacement. Only INSTALLED modules' tools are listed (`collectCliTools` in `module-registry.ts`, unit-tested in `test/cli-tools.test.mjs`).
